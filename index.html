Héritage ≠ Polymorphisme

Héritage Doctrine Polymorphisme Eloquent
Pattern Table Inheritance Polymorphic Associations
But Mapper une hiérarchie de classes Une "FK" qui pointe vers N classes

Avant de commencer, une clarification importante pour ceux qui viennent de Laravel ou Rails.

L'héritage Doctrine et le "polymorphisme" Eloquent (morphTo, morphMany, morphToMany) sont deux patterns complètement différents.

L'héritage Doctrine implémente le pattern "Table Inheritance" de Martin Fowler : on mappe une hiérarchie de classes PHP vers des tables SQL. La colonne discriminante sert à savoir quelle classe PHP instancier.

Le polymorphisme Eloquent, c'est une relation où une colonne *_type stocke le nom du modèle cible, permettant à une table (ex: comments) de pointer vers plusieurs tables différentes (ex: posts, videos). C'est du polymorphisme de relation, pas d'héritage.

Les deux utilisent une colonne discriminante, d'où la confusion fréquente. Mais Doctrine = héritage OOP (is-a, extends), Eloquent = association (has-a).

Aujourd'hui on parle bien d'héritage, pas de relations polymorphiques.

Comment modéliser ça en base de données ?

Nous avons donc une nouvelle classe User qui reprend les propriétés communes (email, password), mais chacun a aussi ses propriétés spécifiques.

Mapped Superclass


#[MappedSuperclass]
abstract class User
{
    #[Id]
    #[GeneratedValue]
    #[Column]
    private ?int $id = null;

    #[Column(length: 255)]
    private string $email;

    #[Column(length: 255)]
    private string $password;
}
							

#[Entity]
class Admin extends User
{
    #[Column(length: 50)]
    private string $role;
}
							

#[Entity]
class Customer extends User
{
    #[Column(type: 'text')]
    private string $address;
}

Doctrine nous offre une première solution : le MappedSuperclass.

On crée une classe abstraite User annotée MappedSuperclass, et nos entités Admin et Customer en héritent (ici j'ai gardé juste une propriété pour l'affichage).

C'est du partage de code, du partage de mapping. C'est simple et ça marche.

Mapped Superclass - SQL


-- Pas de table "user" !
CREATE TABLE admin (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR NOT NULL,
    password VARCHAR NOT NULL,
    role VARCHAR NOT NULL
    permissions JSON NOT NULL
);
CREATE TABLE customer (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR NOT NULL,
    password VARCHAR NOT NULL,
    address TEXT NOT NULL
    loyalty_points INT NOT NULL
);

Avantages : partage de code, simple, pas de NULL

Limites : pas de requête "tous les users"
pas de relation vers User, User n'est pas une entité,

Côté SQL, on obtient 2 tables séparées. Pas de table user.

C'est propre, mais le gros problème : on ne peut pas faire de requête findAll() sur User. Et on ne peut pas avoir de relation ManyToOne vers User non plus, car ce n'est pas une entité.

On veut plus !

  • Requêter tous les Users d'un coup
  • Avoir des relations vers l'entité parente
  • Garder l'héritage en PHP

Il nous faut un vrai héritage en BDD

On veut pouvoir faire un findAll sur User et récupérer des Admin ET des Customer. On veut pouvoir créer des relations vers User.

Pour ça, Doctrine propose deux stratégies d'héritage réel : Single Table et Class Table. Voyons comment ça fonctionne.

Fonctionnement

Le Discriminant

La colonne qui dit à Doctrine quelle classe PHP instancier


#[DiscriminatorColumn(name: 'type', type: 'string')]
#[DiscriminatorMap([
    'admin'    => Admin::class,
    'customer' => Customer::class,
])]

1alice@example.comadmin...
2bob@example.comcustomer...

Avant de voir les stratégies, il faut comprendre le discriminant. C'est LE concept central.

Le DiscriminatorColumn définit la colonne en BDD qui va stocker le type. Le DiscriminatorMap fait le lien entre une valeur dans cette colonne et une classe PHP.

Quand Doctrine lit une ligne avec type='admin', il sait qu'il doit instancier un objet Admin. Idem avec Customer

Single Table Inheritance

(STI)

Première stratégie : le Single Table Inheritance. Toutes les entités dans une seule table.

STI - Code PHP


#[MappedSuperclass]
abstract class User
{
    #[Id]
    #[GeneratedValue]
    #[Column]
    private ?int $id = null;

    #[Column(length: 255)]
    private string $email;

    #[Column(length: 255)]
    private string $password;
}
							

#[Entity]
class Admin extends User
{
    #[Column(length: 50)]
    private string $role;
}
							

#[Entity]
class Customer extends User
{
    #[Column(type: 'text')]
    private string $address;
}

Rien ne change pour les enfants, Admin et Customer étendent User comme avant.

On reprend notre classe User, mais cette fois ce ne sera pas une MappedSuperclass

STI - Code PHP


#[Entity]
#[DiscriminatorColumn(name: 'type', type: 'string')]
#[DiscriminatorMap([
    'admin'    => Admin::class,
    'customer' => Customer::class,
])]
#[InheritanceType('SINGLE_TABLE')]
abstract class User
{
    #[Id]
    #[GeneratedValue]
    #[Column]
    private ?int $id = null;

    #[Column(length: 255)]
    private string $email;

    #[Column(length: 255)]
    private string $password;
}

C'est une vraie Entity. On ajoute le DiscriminatorColumn et le DiscriminatorMap comme vu précédemment, et enfin on introduit InheritanceType SINGLE_TABLE.

Voilà à quoi ressemble notre classe User désormais

STI - SQL généré


CREATE TABLE user (
    id INT AUTO_INCREMENT PRIMARY KEY,
    type VARCHAR(255) NOT NULL,    -- discriminant
    email VARCHAR(255) NOT NULL,
    password VARCHAR(255) NOT NULL,
    role VARCHAR(50),              -- admin, NULL si customer
    permissions JSON,              -- admin, NULL si customer
    address TEXT,                  -- customer, NULL si admin
    loyalty_points INT             -- customer, NULL si admin
);

Côté SQL, tout atterrit dans une seule table. On retrouve les colonnes de notre user, la colonne discriminant 'type', et enfin toutes les colonnes de tous les types.

Vous voyez le problème : les colonnes spécifiques à un type sont NULL pour les autres. Un Admin n'a pas d'adresse, un Customer n'a pas de rôle. Plus on a de types, plus on a de NULL.

Et attention : on ne peut pas mettre de contrainte NOT NULL sur les colonnes des enfants, car elles DOIVENT accepter NULL pour les autres types.

STI - Requêtes


// Tous les utilisateurs (Admin + Customer mélangés)
$users = $em->getRepository(User::class)->findAll();
					

SELECT * FROM user;
-- Doctrine instancie Admin ou Customer selon la colonne "type"
					

// Seulement les Admin
$admins = $em->getRepository(Admin::class)->findAll();
					

SELECT * FROM user WHERE type = 'admin';
					

// Fonctionne aussi en DQL
$query = $em->createQuery(
    'SELECT u FROM App\Entity\User u WHERE u.email LIKE :domain'
);
// Retourne des Admin ET des Customer

Et voilà la magie ! On peut maintenant faire findAll sur User et récupérer des Admin ET des Customer. Doctrine utilise la colonne type pour instancier la bonne classe PHP.

Si on requête directement le repository Admin, Doctrine ajoute automatiquement un WHERE type = 'admin'. C'est transparent.

Le DQL fonctionne aussi : on peut requêter sur User et obtenir un mélange d'entités.

STI - Bilan

Avantages

  • Relations vers User possibles
  • Simple à mettre en place
  • Performances (pas de JOIN)

Inconvénients

  • Table qui grossit avec les sous-types
  • Colonnes avec NULL
  • Pas de NOT NULL sur colonnes enfant
  • Schéma moins lisible

En résumé pour le STI : c'est la stratégie la plus simple et la plus performante car aucun JOIN n'est nécessaire. Par contre, le schéma n'est pas normalisé, on a des NULL partout, et on ne peut pas mettre de contrainte NOT NULL sur les colonnes spécifiques aux enfants.

C'est idéal quand on a peu de propriétés spécifiques par sous-type.

Class Table Inheritance

(CTI)

Deuxième stratégie : le Class Table Inheritance, aussi appelé JOINED. Chaque classe a sa propre table.

CTI - Code PHP


#[Entity]
#[InheritanceType('JOINED')]
#[DiscriminatorColumn(name: 'type', type: 'string')]
#[DiscriminatorMap([
    'admin'    => Admin::class,
    'customer' => Customer::class,
])]
abstract class User
{
    #[Id, GeneratedValue]
    #[Column]
    private ?int $id = null;

    #[Column(length: 255)]
    private string $email;

    #[Column(length: 255)]
    private string $password;
}

Admin et Customer : identiques au STI

Le code PHP est quasi identique au STI. La seule différence : on passe de SINGLE_TABLE à JOINED.

Les classes enfant ne changent pas du tout. C'est la beauté de Doctrine : changer de stratégie ne demande quasiment aucun changement de code PHP.

CTI - SQL généré


CREATE TABLE user (
    id INT AUTO_INCREMENT PRIMARY KEY,
    type VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL,
    password VARCHAR(255) NOT NULL
);

CREATE TABLE admin (
    id INT PRIMARY KEY,
    role VARCHAR(50) NOT NULL,      -- NOT NULL possible !
    permissions JSON NOT NULL,
    FOREIGN KEY (id) REFERENCES user(id)
);

CREATE TABLE customer (
    id INT PRIMARY KEY,
    address TEXT NOT NULL,           -- NOT NULL possible !
    loyalty_points INT NOT NULL,
    FOREIGN KEY (id) REFERENCES user(id)
);

Là c'est très différent côté SQL. On a 3 tables : user avec les colonnes communes, admin et customer avec chacune leurs colonnes spécifiques.

Les tables enfant ont une FK vers user. L'id est partagé : un admin avec id=1 correspond au user avec id=1.

Gros avantage : on peut mettre des NOT NULL sur les colonnes des enfants ! Le schéma est normalisé, propre.

CTI - Requêtes


// Tous les utilisateurs
$users = $em->getRepository(User::class)->findAll();
					

SELECT u.id, u.type, u.email, u.password,
       a.role, a.permissions,
       c.address, c.loyalty_points
FROM user u
LEFT JOIN admin a ON u.id = a.id
LEFT JOIN customer c ON u.id = c.id;
					

// Seulement les Admin
$admins = $em->getRepository(Admin::class)->findAll();
					

SELECT u.id, u.email, u.password,
       a.role, a.permissions
FROM user u
INNER JOIN admin a ON u.id = a.id;

Quand on fait un findAll sur User, Doctrine génère des LEFT JOIN vers toutes les tables enfant. Il a besoin de toutes les données pour instancier le bon objet.

Quand on requête directement Admin, c'est un INNER JOIN : on ne prend que les lignes qui existent dans la table admin.

Plus on a de sous-classes, plus on a de JOINs. C'est là que ça peut devenir coûteux.

CTI - Bilan

Avantages

  • Schéma normalisé
  • Pas de colonnes avec NULL
  • NOT NULL possible sur les enfants
  • Propre et lisible

Inconvénients

  • JOINs systématiques
  • Performances moindres sur gros volumes
  • INSERT = plusieurs tables
  • Plus complexe à débugger

Le CTI est l'inverse du STI : le schéma est propre et normalisé, on peut mettre des contraintes NOT NULL, mais chaque requête implique des JOINs.

Les INSERTs aussi sont plus coûteux car Doctrine doit écrire dans plusieurs tables.

C'est idéal quand on a beaucoup de propriétés spécifiques ou quand l'intégrité du schéma est primordiale.

DQL - INSTANCE OF


// Tous les Admin parmi les Users
$query = $em->createQuery(
    'SELECT u FROM App\Entity\User u
     WHERE u INSTANCE OF App\Entity\Admin'
);
					

// Tous les Users qui ne sont PAS des Admin
$query = $em->createQuery(
    'SELECT u FROM App\Entity\User u
     WHERE u NOT INSTANCE OF App\Entity\Admin'
);
					

// Combiné avec d'autres conditions
$query = $em->createQuery(
    'SELECT u FROM App\Entity\User u
     WHERE u INSTANCE OF App\Entity\Customer
     AND u.loyaltyPoints > :minPoints'
)->setParameter('minPoints', 100);

Le DQL offre un opérateur très puissant : INSTANCE OF. C'est l'équivalent du instanceof PHP, mais côté requête.

Ça permet de filtrer par type directement dans le DQL, sans connaître le nom de la colonne discriminant ni sa valeur. Doctrine traduit ça en WHERE type = 'admin' automatiquement.

On peut aussi l'utiliser en négatif avec NOT INSTANCE OF : par exemple, donner tous les utilisateurs qui ne sont pas admin.

Et bien sûr, on peut le combiner avec n'importe quelle autre condition DQL. Ici on récupère les Customer avec plus de 100 points de fidélité.

DQL - INSTANCE OF


// Dashboard en une seule requête
$query = $em->createQuery(
    'SELECT NEW App\DTO\UserStats(
        COUNT(u),
        SUM(CASE WHEN u INSTANCE OF App\Entity\Admin THEN 1 ELSE 0 END),
        SUM(CASE WHEN u INSTANCE OF App\Entity\Customer THEN 1 ELSE 0 END),
        AVG(c.loyaltyPoints)
    )
    FROM App\Entity\User u
    LEFT JOIN App\Entity\Customer c WITH c.id = u.id'
);
// → UserStats { total: 150, admins: 12, customers: 138, avgLoyalty: 847.5 }

Et le dernier exemple, c'est la cerise sur le gâteau : on combine INSTANCE OF avec CASE WHEN et le mot-clé NEW pour projeter directement dans un DTO. En une seule requête, on obtient un objet UserStats avec le nombre total, le nombre d'admins, de customers, et la moyenne des points de fidélité. Zéro hydratation d'entité, performance maximale, et le résultat est directement un objet typé prêt à utiliser.

C'est valable aussi bien en STI qu'en CTI.

Relations

AuditLog → User


#[Entity]
class AuditLog
{
    #[ManyToOne(
        targetEntity: User::class
    )]
    private User $user;

    #[Column]
    private string $action;
}

Order → Customer


#[Entity]
class Order
{
    #[ManyToOne(
        targetEntity: Customer::class
    )]
    private Customer $customer;

    #[Column]
    private float $total;
}
Une relation vers la classe parente accepte toutes les sous-classes
→ AuditLog.user peut être un Admin ou un Customer

Un des gros avantages de l'héritage Doctrine, c'est le "polymorphisme" (avec des gros guillemets, vs Eloquent) dans les relations. Ici, AuditLog a une ManyToOne vers User : on peut y stocker n'importe quel type d'utilisateur, Admin ou Customer. Tandis que Order pointe spécifiquement vers Customer — seuls les customers peuvent passer des commandes. Doctrine gère la résolution du bon type automatiquement à l'hydratation.

Voilà un récapitulatif. Aucune stratégie n'est parfaite, chacune a ses compromis.

Le MappedSuperclass est le plus simple mais pas de findAll global. Le STI est performant mais avec des NULL. Le CTI est propre mais avec des JOINs.

Le choix dépend de votre contexte. On va justement voir dans la partie 3 des retours d'expérience pour vous aider à choisir.

3.

Retour d'expérience

Quelques pièges qu'on rencontre en production.

Single Table Inheritance

Table de plus en plus large

Colonnes NULL qui s'accumulent

C'est un cas qu'on rencontre tôt ou tard : on a choisi STI, ça marchait bien avec 2-3 sous-types, et puis on a de plus en plus de sous-types. Chaque nouveau sous-type ajoute des colonnes qui seront NULL pour tous les autres.

Migrer de STI vers CTI


-- 1. Créer les tables enfant
CREATE TABLE admin (
    id INT PRIMARY KEY REFERENCES user(id),
    role VARCHAR(50) NOT NULL,
    permissions JSON NOT NULL
);

CREATE TABLE customer (
    id INT PRIMARY KEY REFERENCES user(id),
    address TEXT,
    loyalty_points INT NOT NULL DEFAULT 0
);
					

-- 2. Migrer les données
INSERT INTO admin (id, role, permissions)
SELECT id, role, permissions FROM user WHERE type = 'admin';

INSERT INTO customer (id, address, loyalty_points)
SELECT id, address, loyalty_points FROM user WHERE type = 'customer';

Et on veut passer en CTI.

La migration SQL n'est pas si compliquée : on crée les tables enfant avec une clé primaire qui référence la table parente, puis on migre les données avec des INSERT SELECT.

Migrer de STI vers CTI


-- 3. Nettoyer la table parente
ALTER TABLE user
    DROP COLUMN role,
    DROP COLUMN permissions,
    DROP COLUMN address,
    DROP COLUMN loyalty_points;
					

// 4. Mettre à jour les attributs Doctrine
-#[InheritanceType('SINGLE_TABLE')]
+#[InheritanceType('JOINED')]

⚠ Toujours tester sur un dump de production avant !

Ensuite, on nettoie la table parente en supprimant les colonnes qui ont été déplacées dans les tables enfant.

Côté PHP, c'est un simple changement d'attribut : SINGLE_TABLE devient JOINED. C'est tout.

Mais attention : faites toujours ça sur un dump de prod d'abord. Pas sur la prod directement. Et préparez un rollback. Si la migration plante à mi-chemin, vous voulez pouvoir revenir en arrière.

Pensez aussi à vérifier vos index : en STI vous aviez probablement un index sur la colonne type, en CTI les JOINs sur les clés primaires doivent être performants.

CTI - JOINs coûteux

Avec beaucoup de sous-classes ou de données, les LEFT JOINs ne sont plus si anodins

En CTI, les JOINs systématiques peuvent poser problème sur de gros volumes. Surtout si on a 5-6 sous-classes, chaque findAll fait autant de LEFT JOINs.

Le problème du "downcast"

Un Admin (id: 1) devient un Customer ?


// Ceci ne fonctionne PAS avec Doctrine
$customer = $em->find(Customer::class, 1);
// Ceci non plus
$user = $em->find(User::class, 1);
// Impossible de "transformer" un Admin en Customer via Doctrine

Doctrine ne supporte pas nativement le changement de type

Le discriminant est figé à la création de l'entité

Un piège classique : on ne peut pas changer le type d'une entité. Si un Admin doit redevenir un simple User, Doctrine ne sait pas faire ça nativement.

Le discriminant est écrit une fois à l'INSERT et Doctrine ne fournit pas de mécanisme pour le modifier.

Le problème du "downcast"

Solutions possibles :

  • Requête SQL native pour :
    • modifier le discriminant
    • supprimer l'ancienne ligne enfant
    • insérer la nouvelle ligne enfant
  • Créer une nouvelle entité et supprimer l'ancienne
  • Repenser le modèle (composition plutôt qu'héritage)

On peut le faire en SQL natif : UPDATE le discriminant et DELETE la ligne dans la table enfant. Mais c'est fragile et dangereux.

Mieux vaut souvent repenser le modèle : est-ce que "Admin" est vraiment un type de User, ou plutôt un rôle qu'on peut attribuer et retirer ?

INSTANCE OF et QueryBuilder

⚠ Ne pas utiliser setParameter() avec INSTANCE OF 


// NE PAS FAIRE : setParameter avec INSTANCE OF
$qb->andWhere('u INSTANCE OF :type')
   ->setParameter('type', Admin::class);
// Pas d'erreur explicite, mais résultats incorrects !
					

// Interpoler directement le FQCN dans le DQL
$qb->andWhere(
    sprintf('u INSTANCE OF %s', Admin::class)
);
					

// Ou passer le ClassMetadata via setParameter
$qb->andWhere('u INSTANCE OF :type')
   ->setParameter('type', $em->getClassMetadata(Admin::class));

Un piège remonté par mon équipe pas plus tard que cette semaine : quand on utilise INSTANCE OF dans un QueryBuilder, il ne faut surtout pas passer le FQCN en string via setParameter(). Doctrine ne sait pas interpréter une simple string comme un type d'entité dans ce contexte.

Le pire, c'est qu'il n'y a pas d'erreur explicite : la requête s'exécute, mais ne retourne pas les bonnes données. C'est très difficile à repérer.

Deux solutions : soit on interpole le FQCN directement dans le DQL avec sprintf — c'est l'une des rares exceptions où on n'utilise pas setParameter. Soit on utilise setParameter mais en passant le ClassMetadata de l'entité, récupéré via l'EntityManager. C'est ce que Doctrine attend réellement : un objet metadata, pas une simple string.

Bonnes pratiques

  • Toujours définir le DiscriminatorMap explicitement
    Ne pas laisser Doctrine le deviner (fragile, noms de classes en BDD)
  • Utiliser un Enum pour le discriminant 
    
enum UserType: string {
    case Admin = 'admin';
    case Customer = 'customer';
}

#[DiscriminatorMap([
    UserType::Admin->value    => Admin::class,
    UserType::Customer->value => Customer::class,
])]
    Pas de magic strings
  • Garder la hiérarchie simple - 1 seul niveau d'héritage
  • Indexer la colonne discriminant / id - Accélère les requêtes filtrées
  • Tester les migrations avant de changer de stratégie

Quelques bonnes pratiques tirées de l'expérience.

Toujours définir explicitement le DiscriminatorMap. Si vous ne le faites pas, Doctrine utilise le nom court de la classe en minuscules. C'est fragile : un rename de classe casse la BDD.

Astuce : utiliser un Enum PHP pour les valeurs du discriminant. Ça vous donne le typage fort, l'autocomplétion, et plus de magic strings. Votre classe parente peut exposer un getType qui retourne l'Enum.

Gardez la hiérarchie à un seul niveau. User -> Admin, pas User -> Staff -> Admin. Ça devient vite ingérable.

Pensez à indexer la colonne discriminant, surtout en STI où les requêtes filtrent dessus systématiquement, et les id pour les entités en CTI.

Choisir sa stratégie

Pas besoin de requête globale ?
→ Mapped Superclass - Simple partage de mapping
Peu de propriétés spécifiques par sous-type ?
→ Single Table Inheritance - Performance maximale
Beaucoup de propriétés / besoin de NOT NULL / schéma propre ?
→ Class Table Inheritance - Normalisation
Hiérarchie complexe avec beaucoup de comportements différents ?
→ Repenser le modèle - Peut-être que la composition est plus adaptée

Si vous n'avez pas besoin de requêtes globales, restez sur MappedSuperclass. C'est le plus simple.

Si vos sous-types ont peu de propriétés spécifiques, STI est idéal. Performant, simple.

Si vous avez beaucoup de propriétés spécifiques ou que l'intégrité du schéma est importante, CTI.

Et si votre hiérarchie devient trop complexe, c'est peut-être un signe qu'il faut repenser le modèle. L'héritage n'est pas toujours la réponse, parfois la composition est meilleure.

Composition vs Héritage

Un Admin qui redevient Customer ? Peut-être que le modèle est faux.

Héritage


class User { ... }
class Admin extends User {
    private string $role;
    private array $permissions;
}
class Customer extends User {
    private int $loyaltyPoints;
}

Le type est figé à vie

Composition


class User {
    #[OneToOne]
    private ?AdminProfile $admin;
    #[OneToOne]
    private ?CustomerProfile $customer;
}

Un User peut évoluer librement

On l'a dit dans les bonnes pratiques : parfois la composition est meilleure. Voici un exemple concret.

À gauche, l'héritage : un Admin est un User, pour toujours. Impossible de changer de type, on l'a vu.

À droite, la composition : un User a un profil Admin ET/OU un profil Customer. On peut ajouter ou retirer un profil sans toucher au User. C'est plus flexible.

La composition est meilleure quand le "type" peut changer dans le temps, ou quand un utilisateur peut avoir plusieurs casquettes à la fois.

Cas d'usage concrets (1/2)

  1. User → Admin / Customer
    STI / CTI — selon le nombre de champs spécifiques
  2. Content → Article / Page / BlogPost
    STI — même structure (titre, slug, body), affichage différent
  3. Payment → CardPayment / BankTransfer / Crypto
    CTI — chaque moyen de paiement a ses propres champs
  4. Media → Image / Video / Document
    CTI — métadonnées très différentes
  5. Product → PhysicalProduct / DigitalProduct
    CTI — poids/dimensions vs lien de téléchargement

Voici 10 exemples concrets pour vous inspirer, répartis sur 2 slides.

User/Admin/Customer, on l'a vu tout au long du talk. Les contenus type CMS, c'est un cas classique de STI : Article, Page, BlogPost partagent titre, slug, body et auteur. Seul le rendu et le routing diffèrent.

Les paiements, c'est du CTI car un virement a un IBAN, une carte a un numéro, etc. Beaucoup de champs spécifiques.

Les médias aussi : une image a des dimensions, une vidéo a une durée et un codec, un document a un nombre de pages. CTI est naturel ici.

Les produits physiques vs digitaux, c'est un grand classique e-commerce. Le physique a un poids et des dimensions pour le shipping, le digital a un lien de téléchargement. CTI.

Cas d'usage concrets (2/2)

  1. Event → Conference / Workshop / Meetup
    STI — propriétés communes majoritaires
  2. BaseDocument → Invoice / Quote / CreditNote
    MappedSuperclass — champs communs (n°, date, client), tables séparées
  3. Discount → PercentDiscount / FixedDiscount / FreeShipping
    STI / CTI — logique de calcul différente, peu de champs
  4. Vehicle → Car / Truck / Motorcycle
    CTI — caractéristiques techniques distinctes
  5. AuditLog → LoginLog / PaymentLog / ErrorLog
    STI — table unique pour requêter tout l'historique

Les événements : conférence, workshop, meetup — ils partagent date, lieu, organisateur. STI suffit souvent.

Les documents commerciaux : facture, devis, avoir — ils partagent un numéro, une date, un client et un montant. Mais on ne requête jamais "tous les documents" ensemble. Chaque type vit dans sa propre table → MappedSuperclass idéal.

Les réductions : pourcentage, montant fixe, livraison gratuite. Peu de champs spécifiques, mais la logique de calcul est dans chaque sous-classe. STI / CTI ça se discute.

Les véhicules : une voiture a un nombre de portes, un camion a une charge utile, une moto a une cylindrée. Beaucoup de propriétés distinctes → CTI.

Et enfin les logs d'audit. STI car on veut absolument pouvoir requêter tout l'historique d'un coup, et les champs spécifiques sont minimes.
